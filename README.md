# Travel Order Registry

Standalone system for logging and tracking Travel Orders with **auto-generated numbering**:

```
MM - SEQ - YY
07 - 01  - 26   →  July 2026, 1st travel order that month
07 - 02  - 26   →  July 2026, 2nd
08 - 01  - 26   →  resets to 01 in August
```

The number is generated by a **Postgres trigger** in Supabase (not app code), so it's safe even if two people save a travel order at the same time.

## Stack
- Node.js + Express (backend / API)
- Supabase (Postgres database)
- Vanilla HTML/CSS/JS (frontend, no build step)

## 1. Set up Supabase
1. Create a new Supabase project (or use an existing one).
2. Open **SQL Editor** and run the entire contents of `schema.sql`. This creates the `travel_orders` table plus the auto-numbering trigger.
3. Go to **Project Settings → API** and copy:
   - `Project URL` → `SUPABASE_URL`
   - `service_role` key (NOT the anon key) → `SUPABASE_SERVICE_KEY`

## 2. Run locally
```bash
cd travel-order-tracker
npm install
cp .env.example .env
# edit .env and paste your Supabase URL + service role key
npm start
```
Open `http://localhost:3000`.

## 3. Deploy to Render (same flow as darlu-hub)
1. Push this folder to a new GitHub repo.
2. In Render: **New → Web Service** → connect the repo.
   - Build command: `npm install`
   - Start command: `npm start`
3. Add environment variables in Render's dashboard (Settings → Environment):
   - `SUPABASE_URL`
   - `SUPABASE_SERVICE_KEY`
4. Deploy. Every `git push origin main` will auto-redeploy.

## How the numbering works
- You only pick the **Date of Travel Order** when logging a new entry.
- The trigger reads the month/year from that date, finds the highest `seq_number` already used for that month+year, and adds 1.
- `to_no` is formatted as `MM-SEQ-YY` with zero-padded month/sequence (sequence keeps growing past 99 without truncation — `07-100-26` if it ever gets that busy).
- The TO number is **generated on save** — that's why the "TO Number" field in the form is disabled/preview-only.

## Status flow
Three statuses, matching your original tracker:
1. **Still in the Office** (default)
2. **Disseminated to Region**
3. **Received from Region**

Changing status inline (dropdown in the table) or in the edit modal auto-fills today's date into the matching date column if it's still empty. You can always override the date manually in the edit modal.

## Notes / things you may want to customize
- The `division` field is free-text with autocomplete suggestions (OPARPO, LTS, STOD, SPLIT, FAD) — edit the `<datalist>` in `public/index.html` to match your actual divisions.
- Division colors are auto-assigned by hashing the division name into one of 6 palette colors — consistent every time, no manual color config needed.
- RLS is enabled on `travel_orders` with **no public policies**, so only your backend (using the service role key) can read/write. Safe to leave as-is unless you later want to query Supabase directly from the browser.
